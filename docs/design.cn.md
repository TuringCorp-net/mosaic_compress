# MosaicCompress — 基于自然遗忘曲线的马赛克式对话压缩

> 版本: v0.1.0 | 状态: 设计草案 | 最后更新: 2026-06-08
>
> 关联文档：[L2 Agent 记忆系统设计](original_concept.md#agent-memory)

---

## 一、问题

多轮对话中，LLM 上下文窗口随对话增长而膨胀。传统做法是：

- **Session 管理**：让用户手动"开始新对话" → 丢失历史细节，用户认知负担重
- **滑动窗口截断**：只保留最近 N 轮 → 早期关键信息丢失
- **摘要压缩**：全量历史压缩为一团摘要 → 对话结构丢失，细节不可追溯

三者都有一个共同问题：**用户需要理解并管理 "Session" 这个概念**。对于非技术用户（如作家、创作者），这是不必要的认知负担。

---

## 二、核心思想

> **让 Session 对用户不可见。通过模拟人类"自然遗忘曲线"，实现一个逻辑上永不结束的对话。**

人类的记忆不是"要么全记得，要么全忘记"。越近的事记得越清晰，越远的事越模糊——但重要的事件即使久远，也能留下印象。

MosaicCompress 用大模型自身来模拟这个过程：

- **近期的对话**：保留完整原文（用户最近的交互，细节最重要）
- **稍远的对话**：保留消息骨架，内容做"去水分"蒸馏（Light Compress）
- **更远的对话**：多轮合并为一条叙事摘要（Heavy Compress）
- **极远的对话**：完全移出 LLM 上下文，细节仅保存在 Memory Log 中

对话窗口永不撑爆，用户永远感知不到 Session 的存在。

## 三、参数模型

```typescript
interface MosaicConfig {
  /** 从第几轮开始触发 Light Compress，默认 30 */
  lightStart: number;
  /** Light Compress 每次压缩多少轮，默认 10 */
  lightWindow: number;
  /** 从第几轮开始触发 Heavy Compress，必须 > lightStart，默认 50 */
  heavyStart: number;
  /** Heavy Compress 触发间隔，控制每多少轮将新 Light 块并入 Heavy 块。建议与 lightWindow 一致，默认 10 */
  heavyWindow: number;
  /** 压缩使用的模型，默认 'deepseek-v4-flash' */
  model?: string;
}
```

### 防抖动

压缩不在每轮触发，而是在阈值轮次集中触发：

```
触发条件: 当前轮次 >= lightStart 且 当前轮次 % lightWindow == 0 → Light Compress
触发条件: 当前轮次 >= heavyStart 且 当前轮次 % heavyWindow == 0 → Heavy Compress

建议 heavyWindow = lightWindow，使 Light 和 Heavy 在同一轮次触发，保持结构简洁。
若两者不同，Heavy 触发时将合并多个累积的 Light 块，仍能正常工作但会有临时膨胀。
```

用户感受：每 N 轮对话有一次回复稍慢 ~1-2 秒（大模型在做后台压缩），其余轮次零开销。

## 四、两层压缩

### 4.1 Light Compress（轻量压缩）

**触发**：轮次达到 `lightStart`，且为 `lightWindow` 的整数倍。

**操作**：对目标区间的 `lightWindow` 轮对话，逐条蒸馏内容——**消息数量不变，消息内容蒸馏**。

```
压缩前（2条消息示例）:
  user_1:    "我跟你说一下关于这个世界观的设定啊，我在想魔法体系是不是可以..." (200+ 字)
  assistant_1: "好的，我理解了你的想法。根据你的描述，魔法体系应该采用软魔法..." (800+ 字)

压缩后（仍是2条消息）:
  user_1:    "讨论魔法体系设定，倾向软魔法，关注魔法对角色心理的影响" (~30 字)
  assistant_1: "确认采用软魔法体系：规则模糊但代价明确，不建议走硬核魔法科学路线" (~40 字)
```

**关键设计**：消息的 `role`（user/assistant）不变、顺序不变、条数不变。LLM 仍能看到完整的对话节奏和因果链条——谁说了什么、在什么顺序下说的。

### 4.2 Heavy Compress（重压缩）

**触发**：轮次达到 `heavyStart`，且为 `heavyWindow` 的整数倍。

**首次 Heavy Compress（R = heavyStart）**：将最早被 Light Compress 过的 `heavyWindow` 轮对话（`heavyWindow × 2` 条消息）压缩为 1 条 user + 1 条 assistant 消息。

**后续 Heavy Compress（R > heavyStart）**：**递归合并**——将上一轮 Heavy 的输出（2 条消息）与最新的 Light 块合并，重新压缩为 2 条消息。因此 Heavy 块永远只有 1 个，覆盖范围持续扩大但消息数恒定。

```
首次 Heavy（R=50）:
  输入: Light[1-10: 20条]
  输出: Heavy[1-10: 2条]

递归 Heavy（R=60）:
  输入: Heavy[1-10: 2条] + Light[11-20: 20条]
  输出: Heavy[1-20: 2条]（覆盖范围翻倍）

递归 Heavy（R=70）:
  输入: Heavy[1-20: 2条] + Light[21-30: 20条]
  输出: Heavy[1-30: 2条]
```

**LLM 的自然收敛**：由于输出受 `max_tokens` 约束，且远古信息在反复合并中越来越稀疏，Heavy 块的 token 数不会随覆盖范围无限增长——大模型自动丢弃已被后续对话覆盖的冗余信息，保持摘要精炼。

## 五、行为时间线

以默认配置 `lightStart=30, lightWindow=10, heavyStart=50, heavyWindow=10` 为例：

```
第 1-39 轮:   函数立即返回原数组（未达到压缩阈值）

第 40 轮:     Light Compress(1-10)
              20条消息 → 20条消息（内容蒸馏，条数不变）
              消息总数：80 | 用户感知：回复延迟 +1~2秒

第 41-49 轮:  立即返回，消息数逐步增至 ~98

第 50 轮:     Light Compress(11-20)       → 20条 → 20条
              Heavy Compress(1-10)        → 20条 → 2条（首次，无递归）
              消息总数：82 | 用户感知：回复延迟 +2~3秒

第 51-59 轮:  立即返回，消息数暂时升至 ~100

第 60 轮:     Light Compress(21-30)       → 20条 → 20条
              Heavy Compress(11-20)       → 22条（Heavy[1-10] + Light[11-20]）→ 2条
              消息总数：82（稳态达成） | 用户感知：回复延迟 +2~3秒

第 61+ 轮:    每 10 轮压缩一次，消息数在 82~100 之间周期振荡，永不膨胀。
              第 15,000 轮和第 60 轮的消息总数完全相同。
```

### 第 60 轮后的消息结构（稳态）

```
[0]     system prompt（始终不变，~166K tokens）
[1-2]   Heavy(1-20)       ← 2条（递归合并，覆盖远古至今）
[3-22]  Light(21-30)      ← 20条（最近被 Light 压缩的 10 轮）
[23-82] Raw(31-60)        ← 60条（最近 30 轮完整保留）
[83]    user_61            ← 新消息开始追加...
```

**无论对话进行到多少轮，三层结构始终是 2 + 20 + 60 = 82 条消息。**

## 六、与记忆体系的关系

MosaicCompress 与现有的 L1/L2/L3 记忆体系是**正交互补**的：

| 维度 | L1/L2/L3 记忆体系 | MosaicCompress |
|------|------------------|----------------|
| **做什么** | 从对话中提取持久事实（偏好、决策、画像） | 管理当前对话上下文窗口的信息密度 |
| **输出** | STM/LTM 文件 → 注入 System Prompt Layer 5 | 压缩后的消息块 → 保留在 messages 数组中 |
| **时效** | 跨会话持久 | 在当前"逻辑对话"内 |
| **触发** | Cron 定时 / 条件触发 | 轮次阈值触发 |
| **LLM** | deepseek-v4-flash | deepseek-v4-flash |
| **对用户可见** | 否 | 否 |

两者的配合：

- **MosaicCompress** 保持"当前对话窗口"不膨胀，让 Agent 始终在一个合理大小的上下文内工作
- **STM/LTM** 从对话（包括已被压缩掉的部分）中提炼跨时间的用户画像和关键事实，注入 System Prompt Layer 5
- **Memory Log (L1)** 保存完整原始对话，即使被压缩的内容也可在需要时追溯

## 七、缓存命中分析

MosaicCompress 对 System Prompt 的 DeepSeek 缓存命中几乎没有影响：

```
Token 分布（典型 10 轮对话）:
  messages[0] system prompt:  ~166,000 tokens (94%)
  messages[1..N] 对话历史:     ~10,000 tokens (6%)

当 Light/Heavy Compress 触发时：
  messages[0] 不变 → cache hit (~166K tokens)  ✓
  messages[1..N] 内容压缩 → cache miss (~10K tokens，绝对值相对 System Prompt 极小)

缓存命中率: ~94%，且随着对话轮次增加，命中率反而上升
（因为压缩后用户对话部分的 token 占比进一步缩小）
```

## 八、设计哲学

1. **信任大模型的判断力**：不靠规则截断、不靠 TF-IDF 评分。什么重要、什么可以省略——让 LLM 自己判断
2. **消息骨架不丢**：Light Compress 保留 user/assistant 交替结构，对话的因果链条始终可追踪
3. **自然遗忘，而非暴力截断**：越远的越模糊，越近的越清晰。重要信息即使久远也不会被完全遗忘（通过 STM/LTM 注入 Layer 5 持续提醒）
4. **对用户零感知**：不需要理解 Session，不需要手动管理上下文窗口

## 十、效能分析

> 以下分析基于默认参数：`lightStart=30, lightWindow=10, heavyStart=50, heavyWindow=10`
>
> 中文 token 估算采用 1 字符 ≈ 1 token（保守估计）。

### 10.1 假设

| 指标 | 值 | 说明 |
|------|-----|------|
| 原始用户消息（均值） | ~200 tokens | 中文自然语言，含口语、铺垫 |
| 原始助手消息（均值） | ~800 tokens | 含工具调用说明、创作建议、参考分析 |
| 每轮原始对话（user + assistant） | ~1,000 tokens | — |
| Light 压缩后用户消息 | ~30 tokens | 去口语、留核心意图 |
| Light 压缩后助手消息 | ~80 tokens | 去展开说明、留关键结论 |
| Light 压缩后每轮 | ~110 tokens | — |
| Heavy 压缩后（递归合并 → 2 条消息） | ~1,500 tokens | 跨轮叙事摘要 + 确认记录，受 LLM max_tokens 约束 |

### 10.2 消息数组结构（稳态）

**核心机制**：Heavy Compress 是**递归合并**——每轮 Heavy 的输入包含**上一轮 Heavy 的输出（2 条消息）+ 最新的 Light 块**，合并压缩后仍输出 2 条消息。因此 **Heavy 块永远只有 1 个，不会累积**。

在轮次 R 达到稳态后（R ≥ heavyStart + lightWindow），消息数组（不含 system prompt）分为三层，**条数和 token 数均恒定**：

```
┌──────────────────────────────────────────────────────────────┐
│ Heavy 层: 1 个块 = 2 条消息（约 1,500 tokens）                │
│           覆盖范围：第 1 轮 ~ 第 (R-heavyStart) 轮            │
│           内容：递归合并的跨轮叙事摘要 + 确认记录              │
│           每过 lightWindow 轮，与新 Light 块合并后重新生成     │
├──────────────────────────────────────────────────────────────┤
│ Light 层: 1 个块 = lightWindow 轮                             │
│           每轮 2 条消息（约 110 tokens/轮）                   │
│           总大小恒定：约 2,200 tokens（20 条消息）             │
├──────────────────────────────────────────────────────────────┤
│ Raw 层:   lightStart 轮                                       │
│           每轮 2 条消息（约 1,000 tokens/轮）                 │
│           总大小恒定：约 30,000 tokens（60 条消息）            │
└──────────────────────────────────────────────────────────────┘
```

**总消息数（稳态）= 2 + 2 × lightWindow + 2 × lightStart = 82 条**

在两次压缩之间（如 R=51~59），消息数暂时上升到约 100 条，下一轮压缩时立即回到 82 条。**无论对话进行到第 60 轮还是第 15,000 轮，消息数和 token 数完全相同。**

### 10.3 Heavy 块的递归合并过程

```
R=50: Heavy 输入 = Light[1-10: 20条]
        → Heavy 输出 = 2条（覆盖第 1-10 轮）

R=60: Heavy 输入 = Heavy[1-10: 2条] + Light[11-20: 20条]
        → Heavy 输出 = 2条（覆盖第 1-20 轮）

R=70: Heavy 输入 = Heavy[1-20: 2条] + Light[21-30: 20条]
        → Heavy 输出 = 2条（覆盖第 1-30 轮）

...每 10 轮一次，覆盖范围持续扩大，但始终输出 2 条消息...
```

由于 LLM 输出天然受 `max_tokens` 限制，Heavy 块的 token 数存在物理上限。同时，随着覆盖的远古轮次越来越远，可提炼的新信息越来越少，大模型自然输出更精炼的摘要——**消息数不变，token 数收敛**。

### 10.4 压缩效果对比

| 轮次 | 未压缩 (tokens) | 压缩后 (tokens) | 压缩率 | 消息数 |
|------|----------------|----------------|--------|--------|
| 50 | 50,000 | 32,450 | 35.1% | 82（首次压缩） |
| 60 | 60,000 | 33,700 | 43.8% | 82（稳态达成） |
| 100 | 100,000 | 33,700 | 66.3% | 82 |
| 200 | 200,000 | 33,700 | 83.2% | 82 |
| 500 | 500,000 | 33,700 | 93.3% | 82 |
| 1,000 | 1,000,000 | 33,700 | 96.6% | 82 |
| 5,000 | 5,000,000 | 33,700 | 99.3% | 82 |
| 15,000 | 15,000,000 | 33,700 | 99.8% | 82 |

> 从第 60 轮起，压缩后的 token 数**完全恒定**。第 50 轮为首次压缩的过渡态（Heavy 块覆盖范围仅 10 轮）。压缩率随对话持续自动趋近 100%。

### 10.5 极端场景：每天 50 轮，持续一年

假设重度用户每天 50 轮，连续 365 天（共约 18,000 轮）：

| | 未压缩 | 压缩后 |
|------|--------|--------|
| 消息数（不含 system） | 36,000 条 | **82 条**（恒定） |
| Token 估算 | ~18,000,000 tokens | **~33,700 tokens**（恒定） |
| 加上 System Prompt（~166K） | — | ~199,700 tokens |

**未压缩时**：第 100 轮即超出上下文窗口，对话不可持续。

**压缩后**：从第 60 轮起，上下文大小锁定在 ~34K + 166K = ~200K tokens。**第 18,000 轮和第 60 轮的上下文大小完全相同。对话可以永不结束。**

### 10.6 关键结论

1. **消息数恒定**：稳态 = 82 条，永不增长（仅在两轮压缩之间短暂上升至 ~100 条）
2. **Token 数恒定**：稳态 = ~33,700 tokens（用户对话部分），永不膨胀
3. **压缩率自动趋近 100%**：轮次越多，节省比例越高
4. **上下文窗口永不撑爆**：无论对话持续多久

## 十一、参数约束

仅做一条运行时检查：

```typescript
if (config.heavyStart <= config.lightStart) {
  throw new Error('heavyStart 必须大于 lightStart');
}
```

其余边界场景（如 `heavyWindow < lightWindow`、`lightStart < 1`、压缩区间重叠等）**不做运行时检查**，由文档说明推荐值和约束，使用者自行承担配置选择。
